### D1. 维修描述枚举文案缩写（[Repair.tsx:25-52](src/pages/Repair.tsx:25)）

**根因**：`faultTypeLabels` / `batteryTypeLabels` 的 value 使用缩写（如 `SOC过低报警`、`储能过压报警`），而非 PRD 原文。

**修复**：将 value 改为 Spec §0.1.1/§0.1.2 的 PRD 原文。注意：
- `DC-DC温度` → **DC-DC 温度报警**（DC-DC 后有空格）
- `DC-DC状态` → **DC-DC 状态报警**
- `SOC过高` → **SOC 过高报警**；`SOC跳变` → **SOC 跳变报警**（SOC 后有空格）
- `储能过充` → **车载储能装置类型过充**（无"报警"后缀，PRD 第12项特殊）
- 其余补全：`温差报警`→温度差异报警、`储能过压`→车载储能装置过压报警、`储能欠压`→车载储能装置欠压报警、`单体过压`→单体电池过压报警、`单体欠压`→单体电池欠压报警、`储能不匹配`→可充电储能系统不匹配报警、`单体一致性差`→电池单体一致性差报警

**建议**：抽取为共享常量（如 `src/utils/alertLabels.ts`），供 D1/D2 复用，避免两处维护漂移。

**验证**：
1. 启动 dev server，维修页→新建维修→选"电池类"→打开"维修描述"下拉，逐项核对文案与 PRD §0.1.2 一致
2. 选"故障类"→核对 9 项与 PRD §0.1.1 一致（含 DC-DC 空格）

### D3. 首页预警排行榜未排序（[mock.ts:40-57](src/api/mock.ts:40)）

**根因**：`getAlertRanking()` 直接 `map` 返回，未按 `total` 倒序；`rank=i+1` 取行号而非排名。

**修复**：

```ts
export function getAlertRanking() {
  const tenantVehicles = getFilteredVehicles(false);
  const rows = tenantVehicles.slice(0, 8).map((v, i) => {
    const drive = (i * 7 + 3) % 12;
    const fence = (i * 5 + 1) % 6;
    const fault = (i * 3 + 2) % 4;
    const lowBat = v.soc <= 20 ? 1 : 0;
    return { plate: v.plate, drive, fence, fault, lowBat, total: drive + fence + fault + lowBat };
  });
  return rows
    .sort((a, b) => b.total - a.total)   // 按 total 倒序
    .map((r, i) => ({ rank: i + 1, ...r }));
}
```

**验证**：
1. 首页看板→今日预警排行榜，Total 列应为由高到低递减
2. 序号列 1..n 与 total 排名一致

### D4. DataExport 默认近7天过滤致页面空 + 单测失败（[DataExport.tsx](src/pages/DataExport.tsx) / [mock.ts:611-648](src/api/mock.ts:611) / [VehicleData.integration.test.tsx:60-61](src/__tests__/VehicleData.integration.test.tsx:60)）

**根因**：新增的 `appliedTimeRange` 默认 `[dayjs().subtract(7,'day'), dayjs()]` 过滤，而 mock 导出任务 `createdAt` 全为 06-05/08/09/10（超 7 天窗口），初次加载全部被滤掉 → 页面"No data" + 单测 `getByText('📊 车辆信号数据_V001_20260610.csv')` 失败。

**修复（推荐方案：mock 日期动态化）**：
1. `mock.ts` `_exportTasks` 的 `createdAt` / `expiredAt` 改为相对 `now` 动态生成，使至少 2 条落在近 7 天内：
   - V001：`createdAt = now - 1d`，status=processing
   - V005：`createdAt = now - 3d`，status=completed，`expiredAt = createdAt + 7d`（未过期）
   - V003：`createdAt = now - 10d`，`expiredAt = now - 9d`（已过期，验证过期态）
   - V007：`createdAt = now - 12d`，status=failed
2. 文件名 `车辆信号数据_V001_YYYYMMDDHHmmss.csv` 同步用动态日期生成（PRD 文件名格式即创建时间）。
3. 更新单测断言：不再硬编码 `20260610`，改为正则或部分匹配：
   ```ts
   expect(screen.getByText(/车辆信号数据_V001_\d{8}\.csv/)).toBeDefined();
   expect(screen.getByText(/车辆信号数据_V005_\d{8}\.csv/)).toBeDefined();
   ```
4. 保留 PRD 默认近 7 天窗口不动（合规）。

**备选方案（不推荐）**：回退 `appliedTimeRange` 默认不过滤——但违反 PRD"默认近7天"，仅作降级兜底。

**验证**：
1. `npx vitest run src/__tests__/VehicleData.integration.test.tsx` 全绿
2. dev server → 数据导出记录页，初次加载（默认近7天）应显示 V001/V005 至少 2 条
3. 清空时间范围→显示全部 4 条；选近30天→显示全部

### D5. 驾驶评分公式简化（[mock.ts:226](src/api/mock.ts:226)）

**现状**：`score = Math.max(100 - risks*8, 40)`，单维度近似。
**修复**：按 Spec §1.6.3 实现 5 维度（急加速/急减速/急转弯/AEB 各20%按次数 + 疲劳驾驶20%按时长）映射表，总分四舍五入。mock 可造 5 维度随机次数/时长，套用映射表得 score。
**说明**：mock 数据驱动，无真实信号源，实现映射表逻辑即可证明规则还原。风险等级阈值（≤1/2-3/4-6/≥7）已正确，无需改。
**验证**：驾驶报告→详情，评分与 5 维度次数映射一致（0→100/2→90/4→80/6→70/7-10→60/>10→50；疲劳 0h→100/1h→90…≥5h→50）。

### D6. 围栏弹窗/提示硬编码中文（[Fence.tsx:133-156](src/pages/Fence.tsx:133)）

**现状**：`'请先关闭围栏后再删除'`、`'确认删除'`、`'删除后数据无法恢复，是否继续？'`、`'确认关闭围栏？'`、`'关闭后围栏将停止生效，是否继续？'` 等硬编码。

**验证**：Topbar 切换 EN/ES，围栏删除/关闭确认弹窗文案随之切换。
