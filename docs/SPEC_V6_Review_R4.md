## 一、🔴 高风险问题（P0 — 必须修复）

### R4-H1 · 审计日志数据范围 Spec 内部自相矛盾

- **Spec §1.12.1 `audit_logs`（行 611）**：
  > **数据范围**：审计日志数据范围 = **当前租户 + 所有下级**（与租户管理同，参见 §4.1 数据范围矩阵）。
- **Spec §4.1 数据范围矩阵（行 1026）**：
  > | 系统管理 | 用户/角色/日志 | **仅当前租户** |
- **冲突**：同一份 Spec 内，审计日志数据范围在两处写法完全相反——§1.12.1 说"当前租户+所有下级"，§4.1 说"仅当前租户"。
- **PRD 原文也有内在矛盾**：
  - PRD §数据范围定义（行 171）：`系统管理 | 日志审计 | 仅当前租户`
  - PRD §日志审计·搜索筛选（行 1206）：`所属租户 | 枚举值：当前应用下所有租户名称，附加「全部」`
  - 这说明 PRD 自身"仅当前租户"与"可筛选所有租户"冲突。R1 M16 曾提出此问题，但 V6.3 修订时只在 §1.12.1 改成了"当前租户+所有下级"，**忘记同步修改 §4.1 矩阵**。
- **风险**：🔴 后端实现时取哪个口径？若按 §4.1"仅当前租户"实现，则审计日志"所属租户"筛选下拉虽可显示所有租户名，但查不到下级租户的日志；若按 §1.12.1"当前租户+所有下级"实现，则与 §4.1 矩阵矛盾，且与 PRD §数据范围"仅当前租户"矛盾。
- **建议**：统一为以下口径（需 PM 确认）：
  - 审计日志**查询范围** = 当前租户 + 所有下级（与租户管理同），因为 PRD §搜索筛选明确允许按"所有租户"筛选。
  - §4.1 矩阵中"系统管理-日志审计"改为"**当前租户 + 所有下级**"（与 §1.12.1 一致）。
  - §4.1 矩阵中"系统管理-用户管理/角色管理"保持"仅当前租户"。
  - 或：§4.1 将"系统管理"拆为三行，日志审计单独标注"当前租户+所有下级"。

### R4-H2 · 行程管理"里程报表"独立菜单接口完全缺失

- **PRD 导航栏（行 104）**：
  ```
  | 行程管理 | 里程报表 |  |
  ```
- **PRD §产品概述（行 67）**：
  > 行程管理 → 里程报表：计算每辆车按照日/周/月/年维度，统计里程数据，并通过图表展示出来。
- **PRD §功能权限树（行 282）**：
  ```
  | 行程管理 | 里程报表 |  |
  ```
- **Spec 现状**：
  - §2.5 有 `GET /vehicles/{vin}/mileage`（车辆详情的里程报表 tab，**单车维度**）
  - §2.9 行程管理只有 `GET /trips` 和 `GET /trips/{id}`，**完全没有"里程报表"接口**
  - §4.2 功能权限树也**遗漏了"行程管理→里程报表"**
- **风险**：🔴 行程管理的里程报表是独立菜单项，展示**全租户所有车辆**的里程统计图表（非单车），Spec 完全缺失此接口，前端无法实现该页面。
- **建议**：§2.9 新增 `GET /trips/mileage-report`，参数 `dimension=day|week|month|year` + `plate`(可选)，返回全租户车辆的里程统计数据（与 §2.5 单车接口类似但维度不同：§2.5 是单车的趋势折线，此接口是全租户车辆列表的里程汇总）；§4.2 功能权限树补"行程管理→里程报表"行。

## 二、🟡 中风险问题（P1 — 应修复）

### R4-M1 · §2.1.2 公共 Query 默认 pageSize 与 §0 全局约定冲突

- **Spec §0（行 33）**：
  > 分页 `page`（从 1）+ `pageSize`（可选 10/20/50/100）；**默认值由各页面定义**（车辆列表/围栏/流水类=20；业务管理/系统管理列表类=10）
- **Spec §2.1.2 公共 Query 约定（行 671）**：
  > 分页：`page`, `pageSize`（10/20/50/100，**默认 20**）
- **冲突**：§0 说"默认值由各页面定义"，§2.1.2 又写死"默认 20"。R3-M1 修订了 §0 但**忘记同步修改 §2.1.2**。
- **风险**：🟡 后端若以 §2.1.2 为模板实现公共中间件，所有接口默认 20，业务管理/系统管理列表应为 10。
- **建议**：§2.1.2 改为"分页：`page`, `pageSize`（10/20/50/100）；**默认值由各接口定义，参见 §0**"。

### R4-M2 · assets → vehicles 数据流转链路缺失

- **PRD 业务模型**：
  - `assets` 表（业务管理）= IOT 同步的 VIN 池，划拨到租户
  - `vehicles` 表（车辆管理）= 租户名下的车辆，批量导入时更新
  - 批量导入前提："若系统中存在该 VIN 且**归属当前租户**，执行更新"——即 vehicles 表已有记录
- **Spec 现状**：
  - §1.2.6 `assets` 表有 `tenant_id`（划拨目标）
  - §1.3.1 `vehicles` 表有 `tenant_id` + `vin` PK
  - **但 Spec 从未说明：资产划拨到某租户时，是否自动在 `vehicles` 表创建对应记录？**
- **风险**：🟡 若划拨不自动建 vehicles 记录，则批量导入时"系统中存在该 VIN 且归属当前租户"永远为 false（vehicles 表为空），导入全部跳过。若划拨自动建 vehicles 记录，Spec 未说明字段初始化规则（plate/model 等字段从何而来？）。
- **建议**：§3.3 资产划拨状态机补"划拨到租户 X 时，同步在 `vehicles` 表创建记录（`vin`/`tenant_id`/`device_id` 从 assets 同步，`plate`/`model`/`color` 等扩展字段初始为 NULL，待批量导入填充）"。

### R4-M3 · 驾驶报告"行驶里程"（列表）vs"累计行驶里程"（详情）语义混淆

- **PRD 驾驶报告列表字段**：`行驶里程：单位km`
- **PRD 驾驶报告详情数据指标**：`累计行驶里程：单位km` / `累计行驶时长：单位h`
- **Spec §1.6.2 `driving_reports`**：只有一个 `km` 字段，注释为"累计行驶里程"
- **问题**：列表用"行驶里程"，详情用"累计行驶里程"——两者是否同一数据？PRD 未明确，Spec 也没说。可能的情况：
  - 情况 A：两者都是报告周期内里程，"累计"只是措辞差异 → 一个 `km` 字段即可
  - 情况 B：列表"行驶里程"=报告周期内里程，详情"累计行驶里程"=车辆总里程（vehicles.total_mileage_km）→ 需两个不同字段
- **风险**：🟡 若为情况 B，Spec 只有一个 `km` 字段会导致详情页"累计行驶里程"显示为周期里程而非总里程。
- **建议**：§1.6.2 或 §2.7 注明："列表'行驶里程'与详情'累计行驶里程'均为报告周期内里程，取同一 `km` 字段；详情页如需展示车辆总里程，从 `vehicles.total_mileage_km` JOIN 获取"。或向 PM 确认。

### R4-M4 · 疲劳驾驶"单次连续驾驶"判定逻辑未定义

- **Spec §1.6.3**：
  > 疲劳驾驶时长 = 报告周期内单次连续驾驶超过疲劳阈值（建议 4h）的**累计时长**。
- **问题**："单次连续驾驶"如何从信号计算？
  - "连续"的定义：GPS 持续上报？车辆持续运动（车速>0）？ ignition 信号持续 ON？
  - "单次"的边界：与行程定义一致（GPS≥5min 开始，10min 无定位结束）？还是按点火信号？
  - "累计时长"：报告周期内所有超 4h 的单次驾驶时长之和？还是超 4h 部分之和？
- **风险**：🟡 后端信号处理模块无法实现疲劳驾驶时长统计。
- **建议**：§1.6.3 补"单次连续驾驶 = 车辆持续运动（车速>0 且 GPS 持续上报）的时段，与行程定义对齐（开始=GPS≥5min，结束=10min 无定位）；疲劳驾驶累计时长 = Σ(max(0, 单次连续驾驶时长 - 4h))，即仅累计超阈值部分"。

### R4-M5 · 驾驶报告周期定义缺失（周起始日/月范围）

- **PRD §驾驶报告详情**：
  > 行驶里程趋势 X 轴：周报为**星期一到星期日**；月报为**当月第一天到当月最后一天**
- **Spec §1.6.2**：
  > `period_start` / `period_end` DATE
  > `mileage_trend` JSON 每日里程（折线图：周报 X=周一~周日 / 月报 X=当月每日）
- **问题**：Spec 说了 X 轴是"周一~周日"但没说**周报的 period_start/period_end 如何确定**——是 ISO 周（周一开始）还是其他？月报"当月"是否自然月？跨年如何处理？
- **风险**：🟡 周报 period_seq=21 对应的 period_start/period_end 可能因周起始日定义不同而差一天。
- **建议**：§1.6.2 补"周报按 ISO 8601 周（周一为第一天，周日为最后一天）；月报按自然月（1 日到最后一天）；period_year+period_seq 唯一标识一份报告"。

### R4-M6 · 设备在线/离线 status 更新机制未定义

- **Spec §1.3.2**：
  > status TINYINT 1=在线 0=离线
  > status = (now - last_heartbeat_at) <= 5min ? 1 : 0
- **问题**：`status` 是存储字段还是查询时计算？若是存储字段，谁来更新它？
  - 定时任务扫描？（多久扫一次？）
  - 查询时实时计算？（那 status 字段就是冗余的）
  - 信号上报时更新？
- **风险**：🟡 若 status 不更新，列表显示的"在线/离线"可能是过期数据。
- **建议**：§1.3.2 注明"status 为**查询时实时计算**（`CASE WHEN last_heartbeat_at IS NOT NULL AND TIMESTAMPDIFF(MINUTE, last_heartbeat_at, NOW()) <= 5 THEN 1 ELSE 0 END`），非定时更新存储字段；或定时任务每 1min 扫描更新"。

### R4-M7 · 审计日志 tenantId 参数"全部"传值未定义

- **PRD §日志审计·搜索筛选**：
  > 所属租户 | 下拉选择框 | 枚举值：当前应用下所有租户名称，附加「全部」选项；默认「全部」
- **Spec §2.14**：
  > `GET /sys/audit-log` 参数 `tenantId`
- **问题**：`tenantId` 不传时表示"全部"？还是传 `tenantId=all`？还是传 `tenantId=`（空串）？Spec 未定义。
- **风险**：🟡 前后端"全部"约定不一致导致查询异常。
- **建议**：§2.14 注明"`tenantId` 不传或为空 = 全部（默认）；传具体 ID = 指定租户"。

### R4-M8 · 资产删除后 vehicles 表级联处理未定义

- **PRD §资产划拨·资产删除**：
  > 针对已拆除设备的车辆，进行车辆资产删除操作。
- **Spec §3.3**：
  > 有 device_id 的资产不可删除
- **问题**：PRD 说"已拆除设备"才能删除——即 device_id 应为 NULL（已拆除=设备已解绑）。但 Spec 说"有 device_id 不可删"。逻辑一致。但**删除 assets 记录后，vehicles 表中该 VIN 的记录怎么办？**也删除？保留？
- **风险**：🟡 若 vehicles 保留已删除资产的车，车辆列表会出现"幽灵车辆"（assets 已删但 vehicles 还在）；若不保留，需级联删除规则。
- **建议**：§3.3 补"删除 assets 记录时，同步删除该 VIN 在 `vehicles`/`battery_monitor`/`driving_alerts` 等关联表的数据（物理清理）；或在 vehicles 表增加 `is_asset_deleted` 标记，列表查询时过滤"。

## 三、🟢 低风险问题（P2 — 建议补充）

### R4-L1 · 驾驶评分"取整"规则未定义
- PRD"驾驶评分：取整"，Spec `score INT`。但"取整"是四舍五入（round）还是向下取整（floor）？建议 §1.6.2 注明"四舍五入取整"。

### R4-L2 · 首页实时位置地图接口字段未定义
- §2.15 `GET /dashboard/realtime-map` 只写"实时位置地图数据"。PRD 要求展示"车辆位置信息、车辆信息、电池信息、行驶速度"+"地图自动缩放至展示所有车辆"。建议注明该接口返回 `{vin, lat, lng, plate, soc, speed_kmh}[]` 列表，前端做地图打点+自动缩放。

### R4-L3 · 首页排行榜缺分页与返回字段定义
- §2.15 `GET /dashboard/ranking` 只写"今日预警排行榜"。PRD 列表字段含"序号/车牌/驾驶预警/围栏预警/故障预警/低电预警/总计"。建议注明返回 `{plate, drivingCount, fenceCount, faultCount, lowBatteryCount, total}[]`，按 total 倒序；排行榜不分页（PRD 未提分页，推测全量展示或前端限制条数如 Top 100）。

### R4-L4 · 轨迹回放行程列表接口字段未定义
- §2.12 `GET /monitor/trajectory/{vin}/trips` 只说"行程记录列表"和 duration 格式 HH:mm:ss。PRD 字段含"开始时间/结束时间/开始位置/结束位置/行驶时长/行驶里程"。建议注明返回字段结构与 §2.9 `GET /trips` 的关系（是否复用同一数据源，仅展示格式不同）。

### R4-L5 · 车辆列表/租户列表/用户列表接口返回字段结构未定义
- §2.3 `GET /biz/users`、§2.4 `GET /tenants`、§2.5 `GET /vehicles` 均只写了查询参数，未定义返回字段结构。PRD 对每个列表都给了详细字段清单。建议各接口补"返回字段"说明，至少列字段名，避免前端反复猜。

### R4-L6 · 围栏预警缺 status 字段但联动维修未说明
- §1.5.1 `fence_alerts` 无 `status` 字段——围栏预警不联动维修。PRD §风控预警中围栏预警列表也无"预警状态"列。一致。但 Spec 未明确声明"围栏预警不参与维修联动"，建议 §1.5.1 补一句"围栏预警不联动维修，无 status 字段"。

### R4-L7 · 驾驶预警缺 status 字段说明
- 同上，§1.6.1 `driving_alerts` 无 status 字段——驾驶预警不联动维修。建议 §1.6.1 补一句"驾驶预警不联动维修，无 status 字段"。

### R4-L8 · 语言切换无独立接口声明
- PRD §语言切换"支持中文、英文系统语言切换"。Spec §0 说"Token 内含偏好"但未说如何切换。是 `PUT /auth/language`？还是切换时带 `Accept-Language` Header 由后端更新 Token？建议 §2.2 补一个 `PUT /auth/language {lang: 'zh'|'en'}` 接口或注明"语言切换=前端设置 Accept-Language Header + 后端刷新 Token"。

### R4-L9 · 车辆详情 tab 默认选中未声明
- PRD 列了 7 个 tab（风控预警/驾驶预警/电池监控/充电记录/行程记录/维修记录/里程报表）但未说默认选中哪个。Spec §2.5 也未说。建议 §2.5 注明"默认选中'风控预警记录'tab"（或由 PM 确认）。

### R4-L10 · 总里程/电池快照/实时位置的更新触发机制未声明
- §1.3.1 `total_mileage_km`、§1.7.1 `battery_monitor` 快照、§1.10.1 `vehicle_realtime` 三处都涉及"谁更新这些字段"。Spec 给了表结构但未说更新链路。建议补一句"以上快照/实时表均由**信号处理服务**在接收 OBD 上报时实时更新"。

### R4-L11 · 审计日志操作菜单枚举未列出 13 个一级菜单
- §2.14 `GET /sys/audit-log` 参数 `menu[]` 多选，但未列出枚举值。PRD 说"加载平台全部一级菜单"。建议 §2.14 或 §6.3 补 13 个一级菜单枚举列表（首页看板/车辆管理/实时监控/风控预警/驾驶行为/电池管理/行程管理/围栏管理/维修管理/租户管理/业务管理/车辆数据/系统管理）。

### R4-L12 · 驾驶报告"触发风险数"定义模糊
- PRD 列表字段"触发风险数：触发风险的数量"。Spec §1.6.2 `risk_count INT 触发风险数`。但"风险"指什么？是 5 个评分维度（急加速+急减速+急转弯+疲劳+AEB）的触发次数之和？还是驾驶预警 6 种 ADAS 的次数？两者可能不同。建议 §1.6.2 注明"risk_count = 报告周期内 5 个评分维度触发次数之和（≠驾驶预警 ADAS 次数）"。

## 五、本轮核心结论

1. **Spec 内部自相矛盾是本轮最大发现**：R4-H1 审计日志数据范围在 §1.12.1 和 §4.1 写法完全相反。这是前三轮 Review 的盲区——前三轮都聚焦"Spec vs PRD"对比，忽略了"Spec 内部跨章节一致性"检查。R3-M1 分页默认值修订也遗留了同类问题（§0 改了但 §2.1.2 没同步）。

2. **行程管理里程报表接口完全遗漏**：PRD 导航栏明确有"行程管理→里程报表"独立菜单，Spec 只给了车辆详情的单车里程报表接口，全租户维度的里程报表接口缺失。这是 PRD 业务遗漏。

3. **数据流转链路有多处断点**：assets→vehicles 划拨建表（R4-M2）、设备 status 更新机制（R4-M6）、资产删除级联（R4-M8）、快照表更新链路（R4-L10）——Spec 给了表结构但没给"谁在什么时候更新这些表"的数据流转说明。

4. **疲劳驾驶计算逻辑仍有缺口**：R3-H1 补了评分映射和阈值，但"单次连续驾驶"如何从信号计算（R4-M4）仍然缺失，后端无法落地。

5. **报告周期定义缺失**：周报周几开始（R4-M5）未定义，可能导致 period_seq 与 period_start/period_end 对不上。
