## 一、🔴 高风险（P0 — 新增）

### N-H1 · 电池预警枚举文本与 PRD 不一致（枚举错配，将导致筛选/统计失效）
PRD §电池预警 14 种枚举的**原文文本** vs Spec §5.1 `BatteryAlertType`：

| PRD 原文 | Spec 写法 | 问题 |
|---------|----------|------|
| 温度差异报警 | 温差报警 | 文本不同 |
| SOC低报警 | SOC过低 | 文本不同 |
| 车载储能装置过压报警 | 储能过压 | 缩写 |
| 车载储能装置欠压报警 | 储能欠压 | 缩写 |
| 单体电池过压报警 | 单体过压 | 缩写 |
| 单体电池欠压报警 | 单体欠压 | 缩写 |
| SOC 过高报警 | SOC过高 | 空格 |
| 可充电储能系统不匹配报警 | 储能不匹配 | 缩写 |
| 电池单体一致性差报警 | 单体一致性差 | 缩写 |
| **车载储能装置类型过充** | **储能过充** | **"类型"二字丢失，且与"过充"语义混淆** |
| 充电故障报警 | 充电故障 | 缺"报警" |

- **风险**：🔴 数据库 ENUM、前端 i18n、筛选下拉、首页"低电预警"统计（依赖 SOC低报警）全部对不上；审计文案里的"预警类型"也会错。
- **建议**：Spec §5.1 枚举值**逐字对齐 PRD 原文**；同时统一"首页低电预警"的判定条件为 `type='SOC低报警'`（Spec §1.5.3 现写 `SOC过低`，需改）。

### N-H2 · 故障预警枚举文本与 PRD 不一致（9 种）
PRD §故障预警 9 种枚举原文均带"报警"后缀（VDC故障报警、CDCU故障报警、BDCU故障报警、ADAS故障报警、DC-DC 温度报警、DC-DC 状态报警、驱动电机控制器温度报警、驱动电机温度报警、高压互锁状态报警）。
- Spec §5.1 `FaultType` 写成 `'VDC'|'CDCU'|...`（无"故障报警"后缀）。
- **风险**：🔴 与 N-H1 同类问题，影响列表展示、i18n、审计。
- **建议**：枚举码与展示文案分离——存储用短码 `VDC`，但 Spec 必须补充**展示文案对照表**（`VDC → "VDC故障报警"`），并在 i18n 资源里定义。

### N-H3 · 新建维修"维修描述"是**单选下拉**，不是 TextArea（Spec 写反）
- **PRD §新建维修**：「维修描述：选择故障预警/电池预警对应的**预警类型，单选**」。
- **Spec §5.3 `RepairItem`**：「description 取预警类型名（一键报修）或**手动文本（TextArea，placeholder 按 9 故障/14 电池类型动态提示）**」。
- Spec §6.2 checklist 也写「description…手动文本（TextArea）」。
- **风险**：🔴 实现成 TextArea 与 PRD 严重不符；PRD 是从 9 故障/14 电池枚举里**单选一个**，描述即所选预警类型名称。
- **建议**：① §5.3 `RepairItem.description` 改为「维修类型选定后，从对应枚举（故障 9 种 / 电池 14 种）单选一个预警类型，description = 该类型名」；② §1.9.1 `description` 字段保留 VARCHAR 但来源是枚举值；③ 删除所有"TextArea/placeholder 动态提示"描述。

### N-H4 · 维修状态筛选枚举"已完成" vs 数据库"维修完成"（状态命名冲突）
- **PRD §维修搜索**：维修状态枚举 = `全部 / 维修中 / 已完成`。
- **PRD §维修记录字段**：维修状态 = `维修中 / 维修完成`。
- **Spec §1.9.1** `status` 注释 `1=维修中 2=维修完成`；§2.11 列表筛选 `status` 未给枚举值。
- **风险**：🔴 同一字段在"列表展示"和"筛选下拉"用了两个词（维修完成 vs 已完成），前端/后端/测试极易混淆，且筛选接口不知道传哪个。
- **建议**：Spec 统一术语——**存储码 + 展示文案**：`status: 1=进行中(展示"维修中") / 2=已完成(展示"维修完成")`；筛选接口接收码值，前端两个下拉选项映射到码。明确"列表展示=维修完成""筛选下拉=已完成"是 PRD 故意区分还是笔误，需 PM 确认（Spec 应标注待确认）。

## 二、🟡 中风险（P1 — 新增）

### N-M1 · 一键报修"自动带入车辆数据"未定义带入哪些字段
- PRD：一键报修「自动带入车辆数据」。Spec §3.1 只说"生成 repair(type,desc)"，未列带入字段（vin/plate/预警内容/预警时间？）。
- **建议**：§3.1 明确带入 `{vin, plate, type(故障/电池), description=预警类型名, source_alert_id, source_alert_type}`。

### N-M2 · 维修记录列表"操作人=取最近操作的操作人"语义未定义
- PRD：操作人=「取最近操作的操作人」。一条 repair 的"最近操作"可能是创建、完成、删除前。Spec §1.9.1 `operator_id` 单字段，未说明在"完成维修"时是否更新为完成人。
- **建议**：§1.9.1 注明「operator_id 在创建/完成时更新为当前操作者；删除时不更新（记录已删）」。

### N-M3 · 开通账号"赋予该租户**下级**的管理员角色"——角色归属层级歧义
- PRD §开通账号（账号不存在）：「赋予本用户**该租户下级**的管理员角色」。
- Spec §3.4 写「关联租户+角色」，未说明开通的是**当前租户的管理员角色**还是**为其新建一个下级租户并指派管理员**。
- **风险**：🟡 角色挂错租户层级。
- **建议**：§3.4 明确「开通账号 = 在**当前租户**下创建/复用管理员角色并分配给该用户」；如 PRD"下级"指代不同含义需 PM 确认。

### N-M4 · 驾驶报告"报告时间"格式（周/月）的生成与 i18n 未定义
- PRD：周报示例"2025年第21周"，月报"2025年2月"。Spec §1.6.2 `period_label VARCHAR(32)` 存了字符串，但未定义：① 后端按什么规则生成 label ② 中英文切换时 label 是否重算（"Week 21, 2025" / "Feb 2025"）。
- **建议**：§1.6.2 拆为 `period_year + period_seq`（数值），label 由前端按 locale 拼装；Spec 给出 zh/en 两种格式模板。

### N-M5 · 驾驶报告"风险等级筛选"默认值与可选范围未闭环
- PRD §驾驶报告搜索：风险等级下拉 = `全部/安全/低危/中危/高危`，单选。Spec §2.7 `GET /driving/reports` 参数 `riskLevel` 未列枚举值，也未说默认"全部"。
- **建议**：§2.7 补 `riskLevel ∈ {all,safe,low,mid,high}`，默认 all。

### N-M6 · 充电记录"判定充电"的电流方向符号未与放电统一校验
- PRD：充电电流 **>+2A**，放电电流 **<-2A**。Spec §3.7 正确，但 §1.7.3 `current DECIMAL` 未注明正负号约定（充电正/放电负）。
- **建议**：§1.7.3 注明「current 带符号：充电>0，放电<0」。

### N-M7 · 电池监控"累计充电次数=取充电记录次数"的范围未定义
- PRD：累计充电次数=「车辆接入平台后累计充电次数；首次接入为 0；计算规则=取充电记录次数」。Spec §1.7.1 `charge_count INT` 未说明是否含所有租户历史（车辆划拨后是否累计）。
- **建议**：§1.7.1 注明「按 VIN 全生命周期累计，不随租户划拨重置」。

### N-M8 · 实时位置"车辆列表字段只有 VIN+状态"，但地图气泡字段更多——两接口未区分
- PRD §实时位置：左侧列表字段=VIN+在线/离线；点击车辆气泡=VIN/车牌/企业/设备名/设备状态/车速/上报时间。Spec §2.12 给了 `/monitor/realtime`（列表）和 `/monitor/realtime/{vin}`（详情），但列表接口返回字段未限定为 VIN+状态（可能多返回导致性能问题）。
- **建议**：§2.12 注明列表接口仅返回 `{vin, status}`，详情接口返回完整气泡字段。

### N-M9 · 轨迹回放"默认展示当前全部轨迹，起点=初始启动，终点=最终熄火"算法未定义
- PRD：默认展示当日全部轨迹，起点=车辆移动开始、终点=熄火静止。Spec §2.12 `/monitor/trajectory` 未说明默认时间范围与起终点判定。
- **建议**：§2.12 补默认 `startTime=当日 00:00, endTime=now`；起终点取该范围 tracks 的首末点。

### N-M10 · 审计"操作菜单"多选，但"操作功能"依赖菜单联动——联动查询接口未定义
- PRD §审计搜索：选了"操作菜单"后，"操作功能"仅展示该菜单下的功能。Spec §2.14 有 `/sys/audit-log/actions`，但未说明该接口接收 `menu[]` 参数做联动过滤。
- **建议**：§2.14 `GET /sys/audit-log/actions?menus=业务管理,系统管理` 返回这些菜单下的 action 列表（与 N-相关，第一轮 H6 的枚举定义需含此接口契约）。

## 三、🟢 低风险（P2 — 新增）

### N-L1 · 角色名"长度 10 个字符"的字符计数口径未定义（中文算 1 还是 2）
- PRD：角色名 ≤10 字符。Spec §1.2.5 `name VARCHAR(32)`。中文场景"10 字符"按 UTF-8 字节还是码点？
- **建议**：注明「按 Unicode 码点计，≤10」。

### N-L2 · 围栏名称"最长 30 字符"同上，未定义计数口径
- 同 N-L1，§1.4.1 `name VARCHAR(30)`。

### N-L3 · 用户昵称"长度 20 字符，可重复"未定义计数口径
- §1.1.1 `nickname VARCHAR(64)`，PRD 限 20 字符。

### N-L4 · 登录"账号/密码/验证码均错误→提示账号密码错误"优先级未定义
- PRD 异常表：三种全错时提示"账号密码错误"。Spec §2.2 未定义校验优先级（先验账号密码还是先验验证码）。
- **建议**：§2.2 注明「先校验验证码（错→验证码错误），再校验账号密码（错→账号密码错误）」，与 PRD"三种全错=账号密码错误"一致需调整——实际应按字段顺序，需 PM 确认。

## 六、第二轮核心结论

1. **枚举文本对齐是最大盲区**：第一轮没逐字核对枚举，导致 14 种电池预警 + 9 种故障预警的**展示文案**几乎全部缩写/改写（N-H1/N-H2），这是会导致 i18n、筛选、审计、首页统计**连锁错误**的硬伤。
2. **维修模块三处硬伤**：描述控件类型错（TextArea→单选）、状态命名歧义（维修完成 vs 已完成）、低电预警联动枚举错（N-H3/N-H4 + N-H1）。
3. **"带入/累计/符号"等业务语义**仍有多处空白（N-M1/M2/M6/M7），属实现时会卡住的细节。
4. **建议**：在修订 Spec 前，**先就 N-H4（维修状态术语）和 N-M3（开通账号角色层级）与 PM 确认**，这两项涉及 PRD 自身可能存在的笔误，不宜直接臆断。

> 修订 Spec 时建议新增"枚举对照表"章节（短码 ↔ PRD 原文文案 ↔ i18n key），一次性解决 N-H1/N-H2 及第一轮 H6 的枚举定义问题。
