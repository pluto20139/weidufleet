## 一、🔴 高风险问题（P0 — 必须修复）

### R3-H1 · 疲劳驾驶判定阈值缺失（驾驶评分无法计算）

- **PRD 原文（§驾驶报告·评估维度与指标表）**：
  ```
  | 疲劳驾驶次数 | 单次连续驾驶时长＞**小时 |
  ```
  "＞**小时"中的 `**` 是 **PRD 撰写时的占位符，未填具体数字**。
- **Spec 现状**：§1.6.3 `driving_report_items` 评分表给了疲劳驾驶时长→评分映射（0h→100、1h→90、2h→80…），但**"单次连续驾驶时长超过多少小时算一次疲劳驾驶"这个判定阈值完全缺失**。
- **风险**：🔴 疲劳驾驶占驾驶评分 20% 权重，阈值缺失 → 疲劳驾驶次数无法统计 → 驾驶评分缺一维 → 整份驾驶报告（周报/月报）无法生成 → 风险等级排行榜失效。
- **建议**：① 立即向 PM 确认疲劳驾驶判定阈值（如"单次连续驾驶＞4 小时算 1 次"）；② Spec §1.6.3 补 `fatigue_threshold_hours` 字段并标注 PM 确认值；③ 该阈值应可配置（非硬编码），便于后续调整。

### R3-H2 · 首页"在线/离线"统计口径 PRD 与 Spec 冲突

- **PRD 原文（§首页看板·数据指标）**：
  > 在线数：**OBD设备在线数**
  > 离线数：**OBD设备离线数**
- **Spec 现状（§2.15）**：
  > 在线/离线按"**车辆数**"计（取每车最早安装设备状态，见 §1.3.2）
- **冲突**：PRD 明确按**设备**统计，Spec 改为按**车辆**统计。一辆车装多台设备时，两口径结果不同（如 1 车 3 设备 → 设备口径=3，车辆口径=1）。
- **风险**：🔴 看板数字与 PRD 验收口径不符，UAT 阶段被拒；且 Spec 未声明这是对 PRD 的"修正解读"，也未标 PM 确认。
- **建议**：二选一——
  - **方案 A（推荐）**：Spec 明确标注"PRD 原文为'OBD设备在线数'，本 Spec 解读为按车辆计（一辆车多设备时取判定设备状态），与 §1.3.2 多设备取最早设备规则一致"，并补 PM 确认注。
  - **方案 B**：严格按 PRD，改为按设备数统计（在线数=tenant_id 下 devices.status=1 的设备数）。
  - 无论哪种，需消除 PRD 原文与 Spec 的静默冲突。

## 二、🟡 中风险问题（P1 — 应修复）

### R3-M1 · 分页默认值 PRD 与 Spec 全局规则冲突

- **Spec 现状（§0）**：`分页 page（从 1）+ pageSize（默认 20，可选 10/20/50/100）`——全局统一默认 20。
- **PRD 原文**：多处明确"**默认每页 10 条**"：
  | PRD 位置 | 原文 |
  |---------|------|
  | §业务管理·资产列表 | 默认每页10条 |
  | §业务管理·用户列表 | 默认每页10条 |
  | §业务管理·租户列表 | 默认每页10条 |
  | §系统管理·用户列表 | 默认每页10条 |
- 仅车辆列表、围栏列表、流水页写"默认 20 条"。
- **风险**：🟡 业务管理/系统管理的列表页分页默认值与 PRD 不符，UAT 验收时"翻页后首屏条数"对不上。
- **建议**：Spec §0 分页规则改为"**默认 pageSize 由各页面定义**；车辆列表/围栏/流水类=20，业务管理/系统管理列表类=10"，或在各 API 接口注明 `pageSize 默认值`。

### R3-M2 · 审计映射表声称"47 项"但实际合并后不足 47 行

- **PRD 原文**：§日志审计数据映射表明确列出 **47 行**，序号 1~47。其中：
  - 序号 33 = 风控预警 | 一键报修 | 一键报修（**故障类**）：…
  - 序号 34 = 风控预警 | 一键报修 | 一键报修（**电池类**）：…
  - 序号 38~45 = 系统管理 8 项（新增/编辑/删除用户、重置密码、新增/编辑/删除角色、编辑功能权限）
- **Spec 现状（§5.2）**：
  - 序号 33+34 合并为 1 行：`风控预警 | 一键报修 | 一键报修（{故障/电池}类）：车辆【{v}】，预警类型【{type}】`
  - 序号 38~45 合并为 1 行：`系统管理 | 新增/编辑/删除用户 / 重置密码 / 新增/编辑/删除角色 / 编辑功能权限 | （同业务管理模板，menu=系统管理）`
  - Spec 实际只有约 **39 行**，但头部声称"47 项"。
- **风险**：🟡 ① 审计日志 `action` 字段无法区分"故障类一键报修"vs"电池类一键报修"（两者 action 都是"一键报修"），"操作功能"下拉会少一项；② 系统管理 8 项合并后，`/sys/audit-log/actions` 返回的 `AuditActionEnum[]` 实际不足 47 项，与 §5.2 声称的"47 项"自相矛盾。
- **建议**：§5.2 展开为完整 47 行：① 一键报修拆为两条 action（如 `REPAIR_FAULT` / `REPAIR_BATTERY`，content 模板分别用"故障类"/"电池类"）；② 系统管理 8 项逐行列出（menu=系统管理，action/模板同业务管理但 menu 不同）。

### R3-M3 · 实时位置缺少"企业-车辆层级树"接口

- **PRD 原文（§实时位置）**：
  > 组织层级展示：支持按照**企业层级**展示企业名称、所属车辆数量、所属车辆列表
  > 多选联动：企业、车辆均支持多选。选中车辆后，地图区域会动态展示所选车辆的实时位置
  > 选择企业，则**一键全选**该企业下全部车辆
- **Spec 现状（§2.12）**：
  - `GET /monitor/realtime` → 返回 `{vin, status}` 平铺列表
  - `GET /monitor/realtime/{vin}` → 单车详情气泡
  - **缺少"按企业层级分组"的树结构接口**
- **风险**：🟡 前端无法实现"左侧企业树+车辆列表+多选联动+一键全选企业下车辆"的交互。
- **建议**：§2.12 新增 `GET /monitor/enterprise-vehicles` 接口，返回树结构：
  ```json
  [{ "tenantId": 1, "tenantName": "企业A", "vehicleCount": 50,
     "vehicles": [{"vin":"...", "status":"online"}, ...] }]
  ```
  支持按 tenantId 筛选下级；前端选中企业即全选其下 vehicles[]。

### R3-M4 · 驾驶报告搜索缺"历史1年内"限制和"默认上周/上月"

- **PRD 原文（§驾驶报告·搜索筛选项）**：
  > 周报：按周选择；**仅支持选择历史 1 年内的周**；**默认选择上周**
  > 月报：按月选择；**仅支持选择历史 1 年内的月**；**默认选择上月**
- **Spec 现状（§2.7）**：`GET /driving/reports` 只写 `tab=weekly|monthly / plate / period / riskLevel`，未说明 `period` 的时间范围限制（历史 1 年）和默认值（上周/上月）。
- **风险**：🟡 前端日期选择器无边界约束，用户可选到任意历史周/月；默认值缺失导致首屏无数据。
- **建议**：§2.7 补：① `period` 仅允许选择 `now - 1年` 以内的周/月；② `tab=weekly` 时默认 `period=上周`，`tab=monthly` 时默认 `period=上月`；③ 周报 `period` 格式 `YYYY-Www`（如 2025-W21），月报 `YYYY-MM`。

### R3-M5 · 充电状态二进制编码与枚举未映射

- **PRD 原文**：
  - §充放电记录·充电判定：信号值为 `0100b` / `0011b` 且电流＞+2A 持续 1min → 判定充电
  - §充放电记录·放电判定：信号值为 `0000b` 且电流＜-2A 且车速＞0 持续 1min → 判定放电
  - §0.1.4 充电状态枚举（5 种文字）：未充电 / 准备充电 / 充电中 / 对方放电 / 故障&异常
- **Spec 现状**：§0.1.4 给了 5 种存储码（`NOT_CHARGING`/`PREPARING`/`CHARGING`/`DISCHARGING_OTHER`/`FAULT`），§3.7 给了充放电判定（用 `0100b`/`0011b`/`0000b`），但**二进制编码 → 存储码 → 展示文案三者之间的映射关系完全缺失**。
- **风险**：🟡 后端信号处理模块无法将原始信号值 `0100b` 映射到枚举 `CHARGING`，充放电判定与列表展示脱节；前端"充电状态"列渲染无依据。
- **建议**：§0.1.4 增加一列"信号值编码"，补全映射：
  ```
  0000b → NOT_CHARGING → 未充电
  0011b → PREPARING → 准备充电
  0100b → CHARGING → 充电中
  0110b → DISCHARGING_OTHER → 对方放电  (需 PM 确认编码)
  其他 → FAULT → 故障&异常
  ```

### R3-M6 · 车辆详情"风控预警记录 tab 23 种"数字存疑

- **Spec 现状（§2.5）**：`GET /vehicles/{vin}/alerts` 说明为"风控预警记录 tab（围栏+故障+电池 **23 种**汇总）"。
- **核对**：围栏预警 2 种（出栏/入栏，§0.1.5）+ 故障预警 9 种（§0.1.1）+ 电池预警 14 种（§0.1.2）= **25 种**，非 23。
- **PRD 原文**：§车辆详情·风控预警记录 tab"预警类型：围栏预警+故障预警+电池预警（详见风险预警模块）"——未给具体数字。
- **风险**：🟡 数字错误会导致开发时枚举遗漏或前端筛选下拉项数对不上。
- **建议**：§2.5 将"23 种"修正为"**25 种**（围栏 2 + 故障 9 + 电池 14）"，或若围栏预警不计入"类型"筛选则改为"故障 9 + 电池 14 = 23 种（围栏预警类型单独展示）"并注明。

### R3-M7 · 企业编码长度限制冲突

- **PRD 原文（§租户管理·新增租户）**：企业编码"支持英文和数字，**长度不限制**"。
- **Spec 现状（§1.2.1）**：`code VARCHAR(64)` —— 给了 64 字符上限。
- **风险**：🟡 若实际企业编码超 64 字符，入库时截断或报错，与 PRD"长度不限制"冲突。
- **建议**：§1.2.1 `code` 改为 `VARCHAR(255)` 或 `VARCHAR(128)`，并注明"PRD 原文'长度不限制'，本 Spec 设 255 字符上限为数据库安全约束，超出概率极低"。

### R3-M8 · 里程报表缺各维度展示范围

- **PRD 原文（§车辆详情·里程报表 tab）**：
  > 选择日，展示**近 30 天**每天的里程
  > 选择周，展示**近 30 周**每周的里程
  > 选择月，展示**近 12 个月**每月的里程
  > 选择年，展示**近 3 年**每年的里程
- **Spec 现状（§2.5）**：`GET /vehicles/{vin}/mileage` 只写 `dimension=day|week|month|year`，**未给各维度的默认展示范围**。
- **风险**：🟡 前端折线图 X 轴数据点数量无定义，可能返回全量历史数据导致性能问题，或返回太少导致图表空白。
- **建议**：§2.5 补"各维度默认范围：day→近30天 / week→近30周 / month→近12个月 / year→近3年"。

## 三、🟢 低风险问题（P2 — 建议补充）

### R3-L1 · 轨迹回放缺倍速播放参数声明
- **PRD**："轨迹播放支持倍速：1/2/4/8 倍"。
- **Spec §2.12** `GET /monitor/trajectory` 未提倍速参数。倍速属前端播放器能力（不一定要入接口），但 Spec 应声明"倍速为纯前端播放器能力，轨迹数据接口仅返回点位序列，播放速率由前端控制"。

### R3-L2 · 轨迹回放行程记录时长格式冲突
- **PRD（§轨迹回放·行程记录列表）**："行驶时长：格式**时-分-秒**"。
- **Spec §1.8.1** `duration_min INT 行程时长（HH:mm）`；§0 全局约定"行程时长 HH:mm"。
- 轨迹回放子列表的时长应为 `HH:mm:ss` 而非 `HH:mm`。建议 §2.12 `/monitor/trajectory/{vin}/trips` 注明"该接口 duration 格式为 HH:mm:ss（与行程管理列表 HH:mm 不同，PRD 原文要求）"。

### R3-L3 · 脱敏执行位置 PRD 与 Spec 不一致
- **PRD（§检索匹配规则）**："查询结果返回给前端时，**前端再执行脱敏渲染**"——前端脱敏。
- **Spec §1.14**："后端按请求上下文决定返回脱敏值还是明文（导出接口返回明文）；前端只需渲染"——后端脱敏。
- 两种方案都可行，Spec 方案（后端脱敏）更安全（避免明文泄露到前端 JS 内存），但与 PRD 不一致。建议 §1.14 注明"本 Spec 采用后端脱敏方案（比 PRD'前端脱敏'更安全），导出接口返回明文"。

### R3-L4 · 驾驶报告默认 tab 未声明
- **PRD**："默认展示周报，支持切换成月报"。
- **Spec §2.7** `tab=weekly|monthly` 未说默认 `weekly`。建议补"tab 默认 weekly"。

### R3-L5 · 驾驶评分"1 次"边界未定义
- **PRD 评分表**：次数 0→100、2→90、4→80、6→70、7-10→60、>10→50。
- **1 次**落在哪个区间未给（0 和 2 之间）。Spec §1.6.3 也未补。
- 建议：§1.6.3 补"1 次→归入 0~1 档（100 分），即 n≤1 为 100 分"（与 §0.1.7 安全司机 n≤1 一致）。

### R3-L6 · 电池监控详情缺缺失值插值规则
- **PRD（§电池监控详情）**："如该时刻无上述数据，则取**当天内最接近该时刻的值**插入"。
- **Spec §1.7.2** `battery_consumption_daily` 未说明此插值规则。
- 建议 §1.7.2 补"日放电量/日里程取当日 00:00:00 和 23:59:59 的累计值；若该时刻无数据，取当天内时间戳最接近的值"。

### R3-L7 · 驾驶报告详情缺"车型/外观/车龄"来源说明
- **PRD（§驾驶报告详情）**：字段含车型、外观、车龄。
- **Spec §1.6.2** `driving_reports` 表无这些字段（属 `vehicles` 表）。
- 建议 §2.7 `GET /driving/reports/{id}` 注明"详情接口 JOIN vehicles 表返回 model/color/car_age_years"。

### R3-L8 · 语言默认值未声明
- **PRD（§语言切换）**："默认英文"。
- **Spec §0**："Accept-Language: zh|en"——未声明未指定时默认 `en`。
- 建议 §0 补"未指定 Accept-Language 时默认 en（与 PRD'默认英文'一致）"。

### R3-L9 · 初始密码强制改密拦截时机未明
- **PRD（§修改密码场景1）**："用户每次登录到系统后，自动弹出修改密码弹窗。用户已修改过密码，不再弹出"。
- **Spec §2.2** `/auth/login` 返回 `mustChangePassword` 标志，但未明确该标志为 true 时是**拦截进入系统（必须先改密）**还是**进入后弹窗可关闭**。
- 建议 §2.2 补"mustChangePassword=true 时，前端拦截进入主界面，强制弹出改密弹窗且不可关闭，改密成功后才可进入"。

### R3-L10 · 资产同步增量逻辑未明确
- **PRD（§资产划拨·同步数据）**："首次同步…落库；后续同步…以 VIN 码为唯一标识，匹配已落库的数据，对**增量数据**做入库"。
- **Spec §2.3** `POST /biz/assets/sync` 只写"从 IOT 同步（按 channel_code）"。
- 建议 §2.3 补"增量规则：以 VIN 为唯一标识，已存在则跳过（不更新），不存在则入库；返回 {newCount, skipCount, totalCount}"。

### R3-L11 · 批量资产划拨可选租户层级范围未声明
- **PRD（§批量资产划拨）**："租户树无层级限制，可选择**所有层级内**的租户信息"。
- **Spec §2.3** `POST /biz/assets/batch-transfer` 未说明 `toTenantId` 可选范围。
- 建议 §2.3 补"toTenantId 可选范围=全应用租户树任意层级节点"。

### R3-L12 · 删除角色两种提示文案未给具体文案
- **PRD**：
  - 未关联用户："删除后数据无法恢复，是否继续？"
  - 已关联用户："该角色已关联用户，删除后用户无相关权限，是否继续？"
- **Spec §2.3** `DELETE /biz/roles/{id}` 只写"按是否关联用户给不同提示"。
- 建议 §2.3 补两种文案。

### R3-L13 · 围栏列表缺"使用车辆数"聚合字段
- **PRD（§围栏列表字段）**："使用车辆：使用此围栏的车辆数量"。
- **Spec §1.4.1** `fences` 表无 `vehicle_count` 字段；§2.10 `GET /fences` 列表接口未说明返回聚合的 vehicle_count。
- 建议 §2.10 补"列表接口每项返回 `vehicleCount`（由 fence_vehicles 表 COUNT 聚合，非存储字段）"。

### R3-L14 · 累计充电次数"不随租户划拨重置"为 Spec 解读
- **PRD 原文**："车辆接入平台后累计充电次数…计算规则：取充电记录次数"。
- **Spec §1.7.1** 补充"按 VIN 全生命周期累计，不随租户划拨重置"——这是 Spec 的合理推断（因 charge_records 按 VIN 存），但 PRD"接入平台"语义模糊。
- 建议 §1.7.1 注明"此为 Spec 解读：VIN 全生命周期累计不重置；PRD'接入平台'语义待 PM 确认是否指'首次入库后'"。

### R3-L15 · 驾驶改善建议"兜底"规则未定义
- **Spec §1.6.2** `suggestions JSON 改善建议数组（数据驱动，有兜底）`。
- **PRD** 只说"对每辆车按照周/月根据驾驶行为给出驾驶改善建议"，未定义兜底规则。
- 建议 §1.6.2 补"兜底规则：当某维度无数据（如新车首次报告）时，该项建议显示'本周期暂无{维度}数据，暂无改善建议'"。

### R3-L16 · 租户名称唯一性范围歧义
- **PRD（§新增租户）**："企业名称…唯一性校验，不可重复"——未说是全局唯一还是同父下唯一。
- **Spec §1.2.1** `UNIQUE(name, parent_id, deleted_at)` —— 解读为"同父下唯一"。
- 若 PM 本意是全局唯一，Spec 会允许不同父级下存在同名租户。
- 建议 §1.2.1 注明"此为 Spec 解读：同父下唯一；PRD 未明确范围，待 PM 确认"。

### R3-L17 · 轨迹回放"不支持选择企业"未声明
- **PRD（§轨迹回放）**："仅支持单选车辆；**不支持选择企业**"。
- **Spec §2.12** `GET /monitor/trajectory` 写"vin 必选"但未明确"不支持企业级选择"。
- 建议 §2.12 补"轨迹回放仅支持按车辆单选，不支持按企业多选（与实时位置不同）"。

### R3-L18 · 电池温度计算公式未给
- **PRD（§信号枚举）**："电池温度 = 整车平均温度 = (包1平均 + 包2平均 + 包3平均 + 包4平均 + 包5平均 + 包6平均) ÷ 6"。
- **Spec §6.1** 只写"电池温度(6 包均值)"未给公式。
- 建议 §6.1 补完整公式。

### R3-L19 · 编辑用户邮箱唯一性校验范围未明确
- **PRD（§编辑用户）**："校验登录邮箱是否已被**其他用户**使用"。
- **Spec §2.3** `PUT /biz/users/{userTenantId}` 未明确邮箱唯一性是"全局唯一"还是"该租户下唯一"。
- 实际应为全局唯一（因邮箱是用户中心 users.email UNIQUE）。
- 建议 §2.3 补"编辑用户邮箱校验=全局唯一（users.email），排除当前 user_id"。

### R3-L20 · 首页驾驶风险柱状图默认 range 未声明
- **PRD（§首页看板·数据图表）**："筛选项：今日、近7日、近30日；**默认选择今日**"。
- **Spec §2.15** `GET /dashboard/driving-chart` 写"range=today|7d|30d"未说默认 today。
- 建议 §2.15 补"range 默认 today"。

## 五、本轮核心结论

1. **PRD 自身有一处硬伤未被发现**：R3-H1 疲劳驾驶判定阈值是 PRD 撰写时的占位符"**"，前两轮 Review 均未捕获（因只对比 Spec 与 PRD 文案一致性，未检查 PRD 自身完整性）。这直接阻断驾驶报告生成。

2. **首页"在线/离线"口径是 R1 H2 的遗留争议**：R1 H2 提过"按车辆还是按设备"，Spec V6.2 选择"按车辆"并给出理由，但未标注 PM 确认，且与 PRD 原文"OBD设备在线数"直接冲突。需 PM 一锤定音。

3. **审计映射表"47 项"是数字游戏**：Spec 声称 47 项但实际合并成 39 行，系统管理 8 项缩成 1 行、一键报修 2 项缩成 1 行。这会导致 `/sys/audit-log/actions` 返回的数组长度与"47"不符，前端"操作功能"下拉项数缺失。

4. **实时位置模块缺一个关键接口**：PRD 要求"企业层级树+多选联动+一键全选"，Spec 只给了平铺的 vin 列表接口，前端无法实现 PRD 描述的交互。

5. **分页默认值是"全局统一"思维的代价**：Spec 为了简洁把分页默认值统一为 20，但 PRD 在业务管理/系统管理 4 处明确"默认 10 条"，导致批量冲突。

6. **充电状态二进制编码映射缺失**：PRD 充放电判定用 `0100b`/`0011b`/`0000b`，Spec 枚举用 `CHARGING`/`PREPARING`/`NOT_CHARGING`，两者之间没有映射表，信号处理层无法落地。
