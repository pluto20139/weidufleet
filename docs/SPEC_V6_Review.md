## 一、🔴 高风险问题（P0 — 必须修复）

### H1 · 驾驶报告"风险等级"阈值与 PRD 不一致（数据错算）
- **PRD（§驾驶报告 风险等级划分规则）**：
  - 安全司机：驾驶预警次数 ≤ 1
  - 低危司机：1 < 次数 ≤ 3
  - 中危司机：3 < 次数 ≤ 6
  - 高危司机：> 6
- **Spec 现状**：§5.1 `RiskLevel` 表写成「安全 ≤1 / 低危 1<n≤3 / 中危 3<n≤6 / 高危 >6」——**正确**；但 `PROJECT_CONTEXT.md` 第 243-248 行的历史定义是「低危 2~3 / 中危 4~6 / 高危 >6」，两份文档口径不同，工程实现时易误用 2/4/7 边界。
- **风险**：🔴 评分等级错误，直接影响驾驶报告与排行榜。
- **建议**：Spec 内补一句明确以「≤1 / 2~3 / 4~6 / ≥7」为整数等价表述（PRD 是开区间 `1<n≤3` 即整数 2~3），并废弃 PROJECT_CONTEXT.md 旧定义。

### H2 · 首页"在线/离线"统计口径未定义（指标可能算错）
- **PRD**：「在线数=OBD 设备在线数」「离线数=OBD 设备离线数」——按**设备**统计。
- **Spec 现状**：§1.3.1 `vehicles` 表无设备状态聚合字段；§1.10.1 `vehicle_realtime` 有 `device_status`，但首页 `/dashboard/metrics` 取数逻辑未说明是"按车辆计"还是"按设备计"。一辆车多设备时会产生歧义。
- **风险**：🔴 看板数字与设备列表对不上。
- **建议**：Spec 明确「在线数 = 当前租户车辆中，其判定设备(最早安装设备)在线的数量」，与 §1.3.2 多设备取最早设备的规则一致。

### H3 · 维修"删除回退预警"仅覆盖单一来源，多对一场景未闭环
- **PRD**：删除维修记录后「故障预警、电池预警中此记录的状态回退成"未处理"」。
- **Spec 现状**：§1.9.1 `repairs.source_alert_id` 是单字段；§3.1 状态机假设 1 个 repair ↔ 1 个 alert。但 PRD 的"一键报修"是单条预警生成单条 repair，而**维修记录 tab 的 description 可手动选择预警类型**（§维修管理·新建维修），手动创建的 repair 并无 source alert，回退逻辑应跳过。
- **风险**：🔴 删除手动创建的 repair 时若误回退到某条预警，会造成状态错乱。
- **建议**：Spec §3.1 明确「仅当 `source_alert_id IS NOT NULL`（即来自一键报修）才回退预警；手动创建 repair 删除时不影响任何预警」。

### H4 · 租户"服务到期"逻辑完全缺失（阻断登录后体验）
- **PRD（租户切换）**：
  - 「记录用户最近一次切换的租户，退出登录默认选中最近租户」
  - 「如果最近一次切换的租户**已到期**，则默认选中下一个未到期的租户」
  - 「若登录账户关联下的租户归属的**一级租户服务到期**，则该一级租户及二三级租户显示服务过期」
- **Spec 现状**：§1.2.1 有 `service_expire_at` 字段，但**无任何 API/状态/校验**说明到期如何影响：① 默认租户选择 ② 菜单展示 ③ 业务操作拦截。
- **风险**：🔴 到期租户仍可登录并操作，或登录后白屏。
- **建议**：新增 §3.8「租户服务到期状态机」+ `/auth/my-tenants` 返回每个租户 `isExpired`；登录/切换时若当前租户到期 → 自动切到下一个未到期租户或拒绝进入。

### H5 · 一级租户"仅可添加一个下级"约束未实现
- **PRD（租户层级）**：「一级租户可添加下级租户，**仅可添加一个**。添加后，添加下级入口隐藏」。
- **Spec 现状**：§1.2.1 `tenants` 表与 §2.3 API 均未约束「一级租户下只能挂 1 个二级」。§5.3 `Tenant` 只写了「一级可添加下级」，漏了"仅一个"。
- **风险**：🔴 破坏业务模型，UI"添加下级"按钮无法隐藏。
- **建议**：§1.2.1 增加 CHECK 或应用层校验「parent.level=1 时，该 parent 下 level=2 子节点 ≤1」；API `POST /biz/tenant-tree` 返回 409「一级租户仅可添加一个下级」。

### H6 · 审计日志"操作功能"枚举未定义（日志页筛选无数据源）
- **PRD（日志审计搜索）**：「操作功能」下拉「枚举值根据系统内所有操作功能自动加载；若已选操作菜单，则仅展示该菜单下的功能」。
- **Spec 现状**：§2.14 给了 `GET /sys/audit-log/actions`，但**未定义 47 项 action 的完整枚举与 menu 映射**作为接口返回结构。§5.2 只列了模板，没说这是 actions 接口的返回。
- **风险**：🔴 前端"操作功能"下拉无法实现联动筛选。
- **建议**：Spec 新增枚举对象 `AuditActionEnum: { menu, action, label_zh, label_en }[]`，明确接口返回该数组；`menu` 选中时前端按 menu 过滤。

### H7 · 修改密码"确认新密码失焦校验"未在 API 层闭环
- **PRD**：「确认新密码与新密码不一致时，**输入框失焦时**飘红提示」——前端校验；但服务端必须兜底。
- **Spec 现状**：§2.2 `PUT /auth/change-password` 只说校验"当前密码正确、新旧不同、8~18 位"，**未提"newPassword == confirmPassword"** 服务端校验。
- **风险**：🔴 越过前端直接调接口可设空/不一致确认密码。
- **建议**：§2.2 补充服务端校验项「newPassword === confirmPassword，否则 40002『新密码不一致』」。

### H8 · 批量导入"同文件 VIN 重复仅保留首行"未在 Spec 体现
- **PRD（车辆批量导入）**：「VIN 码重复，同一文件内 VIN 重复，**仅保留首行**；跳过错误数据，生成失败明细」。
- **Spec 现状**：§2.5 只笼统说"VIN 空/非 17 位/租户无此 VIN/同文件重复 → 跳过"，**未明确"保留首行"**的优先级策略，也未说明失败明细里这条重复行如何记录。
- **风险**：🔴 实现时可能"保留最后行"或"全部失败"，与 PRD 不符。
- **建议**：§2.5 明确「按文件行序处理，VIN 首次出现即入库，后续重复行写入 failure_detail（原因=VIN 重复）」。

### H9 · 导出任务"预估行数 >20 万拦截"的预估算法未定义
- **PRD**：「若预估总行数 > 20 万条 → 不创建任务，弹窗拦截」。
- **Spec 现状**：§3.5 说"预估行数 >20 万拦截"，但**未定义如何预估**（count 查询？采样？）。§2.13 `POST /vehicle-data/exports` 也未说明预估接口与创建接口是否分离。
- **风险**：🔴 预估不准会导致误拦截或超大任务压垮服务。
- **建议**：§2.13 拆为两步：① `POST /vehicle-data/exports/estimate` 返回 `{estimatedRows, allowed}`；② 前端 allowed=true 才调创建接口。预估用 `COUNT(*)` 带相同筛选条件。

## 二、🟡 中风险问题（P1 — 应修复）

### M1 · 电池监控"日均电耗"两个公式口径未统一
- **PRD（电池监控列表）**：`日均电耗 = (当日0时累计电耗 − 昨日0时累计电耗) ÷ (当日0时累计里程 − 昨日0时累计里程) × 100`，里程为 0 显示 0。
- **PRD（电池监控详情）**：`日均电耗 = 日放电量 ÷ 日里程 × 100`，单位 kWh/100km，其中日放电量 = 当日 23:59:59 累计耗电 − 当日 00:00:00 累计耗电。
- **Spec 现状**：§1.7.1 用「当日-昨日」口径，§1.7.2 用「当日 0 点 - 当日 23:59」口径，**两处公式不一致但未说明是"列表快照 vs 详情趋势"的差异**。
- **风险**：🟡 同一辆车列表与详情数字对不上，用户困惑。
- **建议**：Spec 明确「列表日均电耗 = 昨日全天口径；详情 30 天趋势 = 每日当天口径」，并注释来源 PRD 段落。

### M2 · 驾驶评分"疲劳驾驶"维度单位混淆（次数 vs 时长）
- **PRD（评分表）**：急加速/急减速/急转弯/AEB 制动按"次数"，疲劳驾驶按"时长(小时)"：0h→100、1h→90、2h→80、3h→70、4h→60、≥5h→50。
- **Spec 现状**：§1.6.3 `driving_report_items` 有 `dimension` 枚举含"疲劳驾驶"，但 `count` 字段同时承载次数和时长，未区分单位。
- **风险**：🟡 评分计算错误。
- **建议**：`driving_report_items` 增加 `unit ENUM('次','时')`；或拆 `count`(次) + `duration_hour`(时) 双字段。

### M3 · 围栏"自定义围栏点位闭合校验"规则缺失
- **PRD**：「支持用户自定义在地图中标记 **99 个点位**并连接闭合」「提交时需校验必填项与围栏点位是否闭合」「支持上一步、清除点位」。
- **Spec 现状**：§1.4.1 `polygon_points JSON` 只说"≤99 闭合"，未定义：① 最少点数（三角形 3 点？）② 闭合判定算法（首尾点相同 vs 自动连回）③ 自定义围栏无 radius/address 时 `radius_km`/`address` 是否强制 NULL。
- **风险**：🟡 可创建非法多边形。
- **建议**：§1.4.1 补「自定义围栏：3 ≤ points ≤ 99，提交时自动闭合（首尾相连），radius/address 强制 NULL」。

### M4 · 车辆"最后位置"地址精度与气泡展示规则未闭环
- **PRD（车辆列表）**：「最后位置：经纬度逆地址解析后的文字描述…**过长时省略显示，点击气泡展示完整地址**」。
- **Spec 现状**：§1.3.1 有 `last_address`，但未定义"省略显示"的截断长度，也未定义"点击气泡"对应哪个接口返回完整地址。
- **风险**：🟡 前端截断长度不统一。
- **建议**：§1.14 补「列表地址截断至 20 字符省略号；气泡/详情接口返回完整 address」。

### M5 · 实时监控"地图缩放聚合"与"分批刷新"逻辑未定义
- **PRD**：「地图比例拉到最大时，**定时分批刷新**车辆数据，具体逻辑参照安驾宝」「车辆数量展示根据**地图缩放做聚合处理**」。
- **Spec 现状**：§1.10.1 仅给字段，未定义聚合算法与刷新机制。
- **风险**：🟡 大车队场景性能差。
- **建议**：§2.12 补「`/monitor/realtime` 支持 `zoom` 参数，按 zoom 层级返回聚合点（cluster）或明细；刷新周期建议 30s」。

### M6 · 轨迹回放"按最早安装设备判定车辆状态"未在 Spec 体现
- **PRD（轨迹回放）**：「若关联多个设备，则按照**关联时间最早**的设备状态来判断车辆状态」。
- **Spec 现状**：§1.3.2 只在 `devices` 说明里提了一句，未在 `vehicle_realtime` 或查询逻辑里落实。
- **风险**：🟡 多设备车辆在线状态判定不一致。
- **建议**：§1.10.1 注明「vehicle_realtime.device_status 取该 vin 最早安装设备的状态」。

### M7 · 业务管理"租户树默认选中根级"与"租户信息页仅根级管理员可见"未闭环
- **PRD（用户管理租户树）**：「页面默认选中根级租户」；**（租户信息）**：「该页面仅面向应用管理员开放（根级租户管理员）」。
- **Spec 现状**：§2.3 未说明 `/biz/tenant-info` 的访问角色限制；§4.4 管理员保护矩阵也未提"租户信息页仅根级可见"。
- **风险**：🟡 越权访问。
- **建议**：§4.4 增加「租户信息页：仅 root tenant 的 admin 角色可访问，其余 403」。

### M8 · 角色"鼠标移入显示编辑/删除图标"的交互与 API 无关但"管理员角色不显示删除"未强调
- **PRD（角色管理）**：「鼠标移入角色，显示编辑、删除图标」「管理员角色**不支持编辑、删除、修改功能权限**」。
- **Spec 现状**：§4.4 已列管理员保护，但未说明 UI 层"管理员行不渲染编辑/删除图标"（而非渲染后置灰）。属实现细节但影响一致性。
- **风险**：🟡 体验偏差。
- **建议**：§4.4 注明「UI 层：管理员角色行不渲染编辑/删除入口（非置灰）」。

### M9 · 删除用户的"仅删该租户关系"语义未在 API 路径体现
- **PRD**：「删除用户，仅删除该用户**在该租户下的对应角色权限关系**，其他租户下的用户不影响」。
- **Spec 现状**：§2.3 `DELETE /biz/users/{userTenantId}` 用 `userTenantId`（即 app_user_tenants.id）语义正确，但 §2.14 系统管理用户删除未强调"仅删当前租户关系"。
- **风险**：🟡 实现可能误删整个用户中心账号。
- **建议**：§2.14 明确「系统管理删用户 = 删 app_user_tenants 该租户行，不删 users 主记录」。

### M10 · 开通账号"账号已存在"分支的"复制信息"内容不同
- **PRD**：
  - 账号不存在 → 复制「登录邮箱 + 初始密码」
  - 账号已存在 → 复制「登录邮箱 + 用户昵称」（**无密码**）
- **Spec 现状**：§3.4 画了分支但未明确"已存在分支不返回密码、复制内容不同"。
- **风险**：🟡 已存在用户误显示密码或泄露。
- **建议**：§3.4 补「账号已存在：返回 {userId,email,nickname}，无 password；复制按钮复制 email+nickname」。

### M11 · 围栏"操作时间"筛选最长 1 年，但"创建时间倒序"与"操作时间"混用
- **PRD（围栏列表）**：「列表数据按照**创建时间倒序**」「操作时间筛选最长支持一年」。
- **Spec 现状**：§1.4.1 有 `created_at`（排序）和 `operated_at`（筛选），字段正确，但未明确"操作时间筛选的是 operated_at 不是 created_at"。
- **风险**：🟡 筛选错字段。
- **建议**：§2.10 注明 `operatedStart/End` 对应 `operated_at`。

### M12 · 行程"预警次数"包含的预警类型范围与首页不一致
- **PRD（行程记录）**：「预警次数：此行程中触发的预警次数，包含**围栏预警、故障预警、电池预警、驾驶预警**」（4 类全含）。
- **Spec 现状**：§1.8.1 `alert_count` 注释正确，但未说明**故障/电池预警处于"维修中/完成"状态是否仍计入**。
- **风险**：🟡 计数口径模糊。
- **建议**：§1.8.1 补「alert_count 统计行程时间区间内所有状态（含已处理）的预警」。

### M13 · 信号数据"同一时间戳成行、缺失显示--"未在表结构体现
- **PRD（车辆信号数据）**：「同一车辆、同一时间戳的所有信号自动成行。若某信号在此时间戳无值，则显示"--"」。
- **Spec 现状**：§1.11.1 是长表（一行一信号），动态列成行需要 pivot，但未说明 pivot 后缺值如何返回（null vs "--"）。
- **风险**：🟡 前端需自行判断 null。
- **建议**：§2.13 注明「pivot 后缺失信号值返回 null，前端渲染为 '--'；接口返回结构以 vin+reported_at 为行键」。

### M14 · 导出文件"过期"状态的判定时机未定义
- **PRD**：「文件有效期 7 天，超时失效，不支持下载」「状态：处理中、已完成、已失败、**已过期**」。
- **Spec 现状**：§1.11.2 有 `expired_at` 与 status=3 已过期，但未说明过期是**定时任务批量更新**还是**下载时实时判定**。
- **风险**：🟡 列表状态显示与实际可下载性不符。
- **建议**：§1.11.2 注明「列表 status 由定时任务每日将 over 7 天的已完成记录置为已过期；下载接口二次校验 now < expired_at」。

### M15 · 仪表盘"今日里程"计算口径未定义
- **PRD**：「今日里程：当前租户名下车辆的**今日里程总数**」。
- **Spec 现状**：§2.15 只说"今日里程"，未说明 = 各车今日 0 点至今的里程增量之和？还是 trips 今日 mileage 之和？
- **风险**：🟡 数字偏差。
- **建议**：§2.15 明确「今日里程 = Σ(车辆 total_mileage 当前值 − 今日 0 点快照值)」。

### M16 · 审计日志"所属租户"筛选项含"全部"但数据范围是"仅当前租户"
- **PRD（数据范围）**：系统管理-日志审计 = **仅当前租户**；但审计搜索「所属租户：枚举值=当前应用下所有租户名称，附加全部」。
- **Spec 现状**：§4.1 写日志=仅当前租户，§2.14 又说 `tenantId` 可选全部应用租户，**两处冲突**。
- **风险**：🟡 越权或筛选失效。
- **建议**：澄清——日志审计**菜单可见性**按当前租户角色权限，但**查询范围**允许看当前租户+下级（与租户管理同），Spec 需统一口径。

## 三、🟢 低风险问题（P2 — 建议补充）

### L1 · 登录"验证码"机制未定义有效期与存储
- Spec §2.2 有 `/auth/captcha` 返回 `captchaId+imageBase64`，但未说明有效期（建议 5min）、存储（Redis）、一次性消费。

### L2 · "复制到剪贴板"的 API 不存在但 Spec 未声明是纯前端
- PRD 多处"复制信息 toast 复制成功"，Spec 未注明这是前端 `navigator.clipboard`，避免后端误实现。

### L3 · 车龄字段 `car_age_years` 是冗余字段，更新策略未说明
- §1.3.1 既存 `purchased_at` 又存 `car_age_years`，应注明「car_age_years 为查询时计算或定时刷新，非用户录入」。

### L4 · 设备"在线/离线"心跳阈值未定义
- §1.3.2 `status` 字段无心跳超时阈值（如 5min 无心跳=离线）。

### L5 · 多语言 i18n 资源结构未定义
- §0 提 `Accept-Language: zh|en`，但未给出错误文案/枚举 label 的双语 key 结构。

### L6 · 软删除 `deleted_at` 与唯一索引冲突
- §1.2.1 `tenants.code UNIQUE` + `deleted_at`：软删后重建同 code 会冲突。建议 UNIQUE 改为 `(code, deleted_at)` 或软删时 code 加后缀。

### L7 · 时区"夏/冬令时切换时刻"未定义精确算法
- §0 说由 dayjs 自动切换，但 PRD 给了"每年 4/9 月第一个星期六 24:00"的精确定义，Spec 应注明依赖 IANA tzdata 的 `America/Santiago` 规则一致。

### L8 · 审计日志"操作结果=失败"的写入时机
- 例如登录失败、删除被拒（有下级的租户）是否记审计？PRD 未明确，Spec 建议补「权限拒绝/校验失败不记审计，仅业务执行失败记」。
